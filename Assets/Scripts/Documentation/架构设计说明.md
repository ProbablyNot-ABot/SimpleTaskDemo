# 项目分层架构说明

## 📐 架构分层

本项目采用清晰的分层架构，职责划分明确：

```
┌─────────────────────────────────────────┐
│          表现层 (Presentation Layer)     │
│  - UI 面板 (DialogPanel, ShopPanel 等)     │
│  - 职责：只负责 UI 显示和用户交互          │
└─────────────────────────────────────────┘
              ↓ 依赖
┌─────────────────────────────────────────┐
│          业务逻辑层 (Business Layer)     │
│  - Manager 类 (DialogManager, NPCManager) │
│  - 职责：处理业务逻辑、流程控制、数据管理  │
└─────────────────────────────────────────┘
              ↓ 依赖
┌─────────────────────────────────────────┐
│          数据层 (Data Layer)             │
│  - ScriptableObject (DialogData, ItemData)│
│  - 职责：存储数据、配置信息               │
└─────────────────────────────────────────┘
```

---

## 🎯 各层职责

### 1️⃣ 表现层 (UI 层)

**成员：**
- `DialogPanel` - 对话面板
- `ShopPanel` - 商店面板
- `BasePanel` - UI 面板基类
- `ItemSlot` - 物品槽

**职责：**
- ✅ 只负责 UI 显示
- ✅ 实现 UI 效果（如打字机效果）
- ✅ 暴露状态和事件供业务层使用
- ✅ 接收数据并显示
- ❌ 不处理业务逻辑
- ❌ 不直接依赖其他 UI 面板

**示例：**
```csharp
// DialogPanel - 只负责显示
public class DialogPanel : BasePanel
{
    public bool IsTyping { get; private set; }  // 暴露状态
    public event Action OnClick;                // 暴露事件
    
    public void Show(string content)            // 接收数据并显示
    {
        StartCoroutine(TypeWriter(content));
    }
    
    public void SkipTyping()                    // 提供操作方法
    {
        // ...
    }
}
```

---

### 2️⃣ 业务逻辑层 (Manager 层)

**成员：**
- `GameManager` - 游戏管理器
- `UIManager` - UI 管理器
- `DialogManager` - 对话管理器
- `NPCManager` - NPC 管理器
- `EventManager` - 事件管理器

**职责：**
- ✅ 处理业务逻辑和流程控制
- ✅ 管理游戏状态
- ✅ 协调不同系统之间的交互
- ✅ 订阅和触发事件
- ✅ 调用 UI 层显示数据
- ❌ 不直接操作 GameObject（除了 Manager 自身）
- ❌ 不存储配置数据（使用 ScriptableObject）

**示例：**
```csharp
// DialogManager - 处理对话业务逻辑
public class DialogManager : MonoBehaviour
{
    public void StartDialog(DialogData data)
    {
        // 1. 打开 UI 面板
        currentPanel = UIManager.Instance.OpenPanel<DialogPanel>("DialogPanel");
        
        // 2. 订阅事件
        currentPanel.OnClick += HandleDialogClick;
        
        // 3. 显示内容
        currentPanel.Show(data.dialogContent);
    }
    
    private void HandleDialogClick()
    {
        // 业务逻辑：判断是跳过还是关闭
        if (currentPanel.IsTyping)
            currentPanel.SkipTyping();
        else
            EndDialog();
    }
}
```

### 3️⃣ 数据层 (Data 层)

**成员：**
- `DialogData` - 对话数据
- `ItemData` - 物品数据
- `ShopData` - 商店数据
- `PlayerData` - 玩家数据
- `EnemyData` - 敌人数据
- `NPC` - NPC 组件（存储引用）

**职责：**
- ✅ 存储配置数据
- ✅ 存储游戏数据
- ✅ 使用 ScriptableObject 便于编辑
- ❌ 不包含逻辑

**示例：**
```csharp
[CreateAssetMenu(fileName = "NewDialogData", menuName = "Data/DialogData")]
public class DialogData : ScriptableObject
{
    public string npcName;
    [TextArea(3, 10)]
    public string dialogContent;
    public List<string> dialogOptions = new List<string>();
}
```

## 📋 各模块详细职责

### GameManager
```
职责：
- 管理玩家数据（PlayerData）
- 管理敌人数据（EnemyData）
- 处理战斗计算（伤害、死亡）
- 管理游戏状态（暂停、恢复）
- 处理游戏结束逻辑

职责外：
- ❌ 不处理 UI 显示
- ❌ 不处理玩家移动
- ❌ 不处理对话
```

### UIManager
```
职责：
- UI 面板的创建、缓存、销毁
- 管理 UI 层级关系
- 提供面板打开/关闭接口
- 关闭所有面板

职责外：
- ❌ 不处理面板内部逻辑
- ❌ 不处理业务逻辑
- ❌ 不存储面板数据
```

### DialogManager
```
职责：
- 管理对话流程（开始、翻页、结束）
- 处理对话中的用户交互
- 管理对话数据
- 触发对话相关事件

职责外：
- ❌ 不处理 UI 显示效果
- ❌ 不存储对话配置数据
- ❌ 不处理 NPC 管理
```

### NPCManager
```
职责：
- 管理场景中所有 NPC
- NPC 的注册和注销
- 提供 NPC 查询接口
- 管理 NPC 交互状态

职责外：
- ❌ 不处理对话逻辑
- ❌ 不处理 NPC AI
- ❌ 不处理玩家移动
```

### PlayerMove
```
职责：
- 处理玩家输入
- 控制玩家移动
- 处理寻路
- 触发 NPC 交互

职责外：
- ❌ 不处理对话逻辑
- ❌ 不处理战斗逻辑
- ❌ 不处理 UI 逻辑
```

## 🔄 数据流向

### 对话系统数据流
```
NPC (Data 层)
  ↓ 提供数据
DialogManager (业务层)
  ↓ 调用
UIManager (业务层)
  ↓ 创建
DialogPanel (表现层)
  ↓ 显示
玩家看到对话内容
```

### 用户交互数据流
```
玩家点击 (Input)
  ↓
DialogPanel (表现层)
  ↓ 触发事件
DialogManager (业务层)
  ↓ 处理逻辑
DialogPanel (表现层)
  ↓ 更新显示
玩家看到结果
```

## 🎯 架构优势

### 1. 职责清晰
- 每个类只做一件事
- 代码易于理解和维护

### 2. 低耦合
- 层与层之间通过接口通信
- 修改一层不影响其他层

### 3. 高复用
- UI 面板可以在任何地方使用
- Manager 可以被多个系统调用
- 数据可以共享

### 4. 易扩展
- 添加新功能只需修改对应层
- 不影响现有代码

### 5. 易测试
- 业务逻辑集中在 Manager，易于单元测试
- UI 层可以独立测试显示效果

## 📝 使用示例

### 打开对话框
```csharp
// PlayerMove 中触发
void OnReachNPC()
{
    var npc = currentNpc.GetComponent<NPC>();
    if (npc != null && npc.dialogData != null)
    {
        // 只调用 Manager，不处理具体逻辑
        DialogManager.Instance.StartDialog(npc.dialogData);
    }
}
```

### 打开商店
```csharp
// 任何地方都可以
UIManager.Instance.OpenPanel<ShopPanel>("ShopPanel");
```

### 暂停游戏
```csharp
GameManager.Instance.PauseGame();
```

## 📌 总结

本架构遵循以下原则：

1. **单一职责原则** - 每个类只做一件事
2. **开闭原则** - 对扩展开放，对修改关闭
3. **依赖倒置原则** - 依赖抽象不依赖具体
4. **接口隔离原则** - 使用细粒度的接口
5. **分层架构原则** - 清晰的层次划分

这样的架构使得项目：
- ✅ 易于理解
- ✅ 易于维护
- ✅ 易于扩展
- ✅ 易于测试
- ✅ 代码复用性高
